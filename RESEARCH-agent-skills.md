# 從 addyosmani/agent-skills 學到什麼 —— loops-workflow 借鑑建議

> 我把 `addyosmani/agent-skills` 這個 repo 整包 clone 下來**實際讀過**(不是只用 web 搜),它有 26 個 skill、8 個 slash command、4 個 reviewer persona、一份 orchestration 規範。下面挑出值得搬進我們 loops-workflow 的東西,還有幾個要你拍板的岔路。我啟動的 deep research(web 二手綜合)還在背景跑,回來會再幫這些判斷做一次交叉驗證。

## 一、值得納入的好東西（白話版）

### 1. 「另派一個人專門挑你毛病，而且不告訴他你的答案」＋「繞圈最多 3 圈就停」

**它在幹嘛**：它有個招數叫 `doubt-driven-development`，白話就是「重要的決定別自己說了算，另外叫一個全新的腦袋來專門挑錯」。最關鍵的一條規矩是 —— **挑錯的人只給他『東西』和『要求』，故意不給他你覺得對的結論**，免得他順著你點頭。它還規定「挑錯→修正」這種繞圈**最多繞 3 圈**，還沒好就停下來問人，不准無限繞。

**補我們什麼**：
- 我們 `verify` 階段本來就要派一個 reviewer 反偏見，但之前只寫「要挑錯」，沒給它具體該怎麼挑。現在可以直接用它那套 adversarial 腳本（「找出哪裡錯，不要附和、不要總結」）。
- 我們 `iterate` 回環之前**沒講最多繞幾圈** —— 這是個漏洞，可能無限迴圈。直接補上「3 圈上限，超過就 escalate 給你」。

**動到**：`verify`、`iterate`
**我的看法**：很推薦。這直接補上我們「回環停止條件」這個缺口。

---

### 2. 每個 skill 都附「偷懶藉口 vs 反駁」表 ＋ 結尾「沒證據不准說完工」檢查表

**它在幹嘛**：它每個 skill 結尾固定有兩張表 ——
- 一張列「AI 最常用來跳步驟的藉口」配上反駁（例如藉口「測試晚點再補」→ 反駁「晚點＝永遠不補」）。
- 一張「要拿出什麼證據才算做完」的勾選清單（測試綠了、build 過了、跑起來的畫面…）。

**補我們什麼**：讓我們 7 個 skill 不會被 AI 自己找藉口跳掉關鍵步驟，而且「這階段算不算做完」有客觀證據，不是「看起來差不多」。

**動到**：全部 7 個 skill 的骨架
**我的看法**：很推薦，跟你原本繁中 house style 不衝突，是加料不是改味。

---

### 3. 「要餵給 AI 的資料分 5 層，一個任務別塞超過 2000 行」

**它在幹嘛**：它把「給 AI 的上下文」分成 5 層（專案規則 → 規格 → 相關檔案 → 錯誤訊息 → 對話歷史），而且給了**具體數字**：一個任務塞給 AI 的東西盡量**少於 2000 行**，理由是「視窗開很大，不代表 AI 的注意力也那麼大」。

**補我們什麼**：我們之前講「主線只留決策軌跡、別吸收原始素材」是個**原則**，但沒有可衡量的標準。現在有個數字當目標，`.loops/` 每份檔案、每次餵給階段的量都可以對著這個數字控制。

**動到**：`.loops/` 設計（§7）、主線 context 控制（§6.5）
**我的看法**：推薦。把抽象原則變成可檢查的數字。

---

### 4. 「找資料就用內建的 Explore，別自己造」＋ build 要有一張「遇到這些就停下來問人」的清單

**它在幹嘛**：
- (a) 它說探索 / 找資料就用 Claude Code **內建的 `Explore`**（它跑便宜的 Haiku、而且被禁止亂改檔案，天生就是幹這個的），不用自己另外定義一個探索 agent。
- (b) 它的 build 列了一張**「該停下來問人」的清單**：測試怎樣都弄不綠、規格講不清楚、或要做**危險又不可逆的事**（改權限、搬資料庫、付款、刪資料、任何 `git revert` 救不回來的）—— 碰到就停，不准硬幹。

**補我們什麼**：`explore` 直接用內建的，省成本又省得自己維護；`build` 多一道明確的安全護欄，AI 不會悶著頭做危險操作。

**動到**：`explore`、`build`（§6.5）
**我的看法**：推薦。兩個都是「免費的安全升級」。

## 二、要你拍板的岔路

### 岔路 1（重要）：我們的 `dispatch`，它說是反模式

**怎麼回事**：它有條很硬的規矩 ——「**不要做一個專門幫你決定該呼叫哪個指令的東西**」，它管這叫 router，明白列為反模式。它的理由是：
- 這層純粹轉手、本身沒幹實事；
- 多繞一圈會**多花大約一倍 token**；
- 你其實早就知道自己要幹嘛，直接喊那個指令就好。

我們的 `dispatch`（判斷你這次是 issue / 設計 / 修正，再帶你到對應起點）正是在做這件事。

**但我覺得我們有一半可以豁免**：
- 這是**你自己要的**「一行指令判斷類型」需求，不是我硬加的轉手層；
- 它真正在罵的是那種「一路自己接著做、把人晾在旁邊」的 router。我們 `dispatch` 很薄，只做「分類 + 建 `.loops/` 狀態檔 + 建議從哪開始 + 交棒」，交棒後每個階段都會停下來等你 —— 沒踩到它的真痛點；
- 三種類型切入的**起點階段不一樣**（issue 從 goal、設計從 explore、修正從 iterate），這個分類有實際用處，不是純轉手。

**二手驗證補了關鍵一筆**：我另外跑的 deep research 發現 —— addyosmani **自己就有一個叫 `using-agent-skills` 的「總機 skill」**，用一棵「遇到 X → 用 Y」的決策樹把工作分流到單一 skill。換句話說，他們罵的 router 反模式，罵的是「persona 互相亂呼叫 + 自動 paraphrase 串接、把人晾在旁邊」那種；**用一棵清楚的決策樹分流、分完就交給人**，他們自己天天在做、完全正派。這把天平更明確推向「保留 dispatch」。

**我的傾向**：**保留 `dispatch`，做成一棵「遇到 X → 從 Y 階段開始」的決策樹、但明文做薄**（只准分類、建狀態、建議起點、交棒，禁止自己一路 paraphrase 串接），**同時學它的好做法 —— 在 plugin 的 `AGENTS.md` 寫一張「什麼情況用哪個指令」的對照表**，這樣你也能跳過 dispatch 直接喊階段，或讓系統靠描述自動觸發對的階段。

> 依你「外部參考只有參考價值、決策權在你」的原則，這只是我的一票傾向加一票佐證，最後你決定。等下會問你三個選項：保留做薄（我推薦）/ 拿掉改純 AGENTS.md / 原樣不動。

---

### 岔路 2（順帶提）：要不要替「自動連跑」留一道未來的門

**怎麼回事**：它的 build 有個 `auto` 開關 ——「你核准一次計畫，它就一路把所有任務做完，但**每個任務照樣寫測試、各自 commit，遇到失敗或危險操作還是會停**」。這跟你之前選的純 Closed Loop（每階段都停）不一樣，但它是個**有保留驗證的折衷**，不是放生亂跑。

你上次在「推進模式」那題選了純 Closed Loop，我尊重。這裡只是讓你知道：如果哪天你嫌每階段都停太囉嗦，這個「核准一次、危險才停」的 auto 模式是業界驗證過的安全版本，可以當未來的門。**這次先不做，列為未來考量**，除非你現在就想加。

## 補充：deep research 二手驗證（交叉檢查過了）

我另外跑的 deep research（掃 21 個來源、抽 104 條主張、3 票對抗式驗證留下 20 條）回來了，跟我親手讀 repo 的結論**一致**，再補幾點：

**又多印證的**：
- 「派獨立的人挑錯、不給他你的結論」他們有獨立一個 skill（`doubt-driven`）在做，跟我們紅綠分離同源 —— 二手再次確認。
- 「每個 skill 結尾要有證據才算完工」「description 要寫 what + when」連 Anthropic 官方文件都佐證。

**新補的好東西**：
1. **dispatch 有正派範本**：addyosmani 的 `using-agent-skills` 就是用決策樹分流的「總機」（見岔路 1）。我們 dispatch 做成決策樹就站得住腳。
2. **並行派 agent 有張 4 項安全檢查表**：① 能同時跑不打架 ② 每個 agent 找的是**不同種類**的問題 ③ 結果併回主線塞得下 ④ 等待時間夠長值得並行。explore / verify 並行前先過這張表，而且要「**同一個回合一次發多個 agent**」才會真的並行。
3. **description 千萬別把步驟寫進去**：不然 AI 會照 description 的摘要做、不去讀完整 skill，等於短路了我們的 human gate。
4. **Anthropic 官方還有一招**：先產出「結構化、可被腳本檢查的計畫檔」，腳本驗過才執行（plan → validate → execute）。我們 `.loops/` 現在是純 markdown 給人看；未來可考慮讓 plan 檔有一塊「機器可驗證」的結構，但這是進階，先不做。

**誠實提醒（驗證時被打折的）**：
- 「subagent 絕對不能再派 subagent」這句的**絕對措辭**在驗證時被投反對（可能某些來源沒講死）。但**我們設計本來就不靠巢狀**（都是主線派），所以對我們沒影響。
- 他們是**線性**流程（做完就 ship），我們是**繞圈**（有 explore / iterate 他們沒有）—— 所以**別逐格照抄他們的階段**，只借觀念。

**它丟回 4 個好問題（我的初步答覆）**：
- dispatch 是自動跑完還是只分流 + 停？→ 我們已定：只分流 + 停（Closed Loop）。
- `.loops/` 要不要升級成腳本可驗證的計畫檔？→ 進階，先不做、記著。
- 紅綠分離要不要像 doubt-driven 一樣「連結論都不給 reviewer」？→ 可強化 verify：reviewer 別看「作者說已通過」。
- 要不要建一套評估場景（≥3 + baseline）？→ 進階，看之後要不要。
