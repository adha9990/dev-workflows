# 驗收審查（acceptance-review）

> product-contract-reviewer 用：把 issue / 完工定義當契約、把改動當履約證據，逐條對照。重點不只「有沒有超出範圍」，更是「**有沒有在解對的問題、是不是真的完成**」。

## 一、錯題偵測（最高級，P0）

先確認整個改動**在解對的問題**：

- 主要行為是否對上 issue 的核心目標？
- 大量改動是否落在**無關的問題**上（順手改了別的東西、把一張 issue 變成多個問題一起改）？
- 額外 scope 是否模糊化了原本要交付的東西？

主行為沒對上目標 = 做錯了問題，P0；細節做再多都白費。

## 二、完成度五態

每條 acceptance criterion 判一個狀態，不要只有「有 / 沒有」二分：

| 狀態 | 意思 |
|------|------|
| **已滿足** | 有實作 + 有可信證據 |
| **部分滿足** | 某條路徑能 work，但缺某個狀態 / 入口 / 持久化 —— 在 merge-readiness 下**通常仍算未完成** |
| **缺失** | 沒做 |
| **證據不足** | 看起來像有做，但無法證明（缺驗證證據） |
| **被反證** | 當前行為與要求矛盾 |

partial 要寫清楚：哪條路徑能 work、缺哪個狀態 / 入口 / 持久化。

## 三、端到端鏈路 + 持久化驗證

- **只完成片段不算履約**：要驗整條鏈 `使用者動作 → 傳輸 → 服務 → 持久化 → 回顯` 都通；只證 UI 暫時變了、或只證後端有對應入口，都是 partial。
- **持久化要真**：凡 issue 要求保存 / 提交 / 編輯持久化，要驗**重新載入 / 重新查詢後結果仍在**；只有樂觀更新或快取更新**不算**持久化證據。
- 涉及非同步行為時，逐一對照狀態全集：開始 / 進行中 / 成功 / 失敗 / 空結果 / 重複觸發 / 取消 / 重入，是否都有對應實作。

## 四、驗證證據強弱分級

判斷 PR / 改動附的證據夠不夠強：

- **弱證據（不足以證明核心行為）**：「本地測過」沒附步驟、只貼一句 pass 看不出測了哪條、沒有前後操作的截圖、純從 code 推斷。
- **核心使用者行為**優先要至少一份**可執行**證據（測試 / 可重現步驟），不能只靠 code 推斷或弱證據。
- 跨端 / 跨層的行為，優先要求端到端可執行的驗證。

## 五、Finding 寫法

對齊雙視角：先工程視角（缺哪條 criterion / 哪條鏈斷在哪），再使用者視角（使用者會發現什麼沒做到）。partial / 證據不足 / 被反證都要明確標出，不含糊帶過。
