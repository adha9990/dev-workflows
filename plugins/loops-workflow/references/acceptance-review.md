# 驗收審查（acceptance-review）

> product-contract-reviewer 用：把 issue / 完工定義當契約、把改動當履約證據，逐條對照。重點不只「有沒有超出範圍」，更是「**有沒有在解對的問題、是不是真的完成**」。

## 一、錯題偵測（最高級，P0）

先確認整個改動**在解對的問題**：

- 主要行為是否對上 issue 的核心目標？
- 大量改動是否落在**無關的問題**上（順手改了別的東西、把一張 issue 變成多個問題一起改）？
- 額外 scope 是否模糊化了原本要交付的東西？

主行為沒對上目標 = 做錯了問題，P0；細節做再多都白費。

## 二、完成度五態（**逐項列舉、強制完整**）

**把 issue 的 acceptance criterion 逐條抽出、每一條都判一個狀態** —— 不是挑幾條講、不是只有「有 / 沒有」二分。**列舉必須完整**：issue 寫的每件事（含散在描述 / 舉例 / 非目標裡的隱含需求）都要在 ledger 出現一條，沒著落 = 漏列。

| 狀態 | 意思 |
|------|------|
| **已滿足** | 有實作 + 有可信證據 |
| **部分滿足** | 某條路徑能 work，但缺某個狀態 / 入口 / 持久化 —— 在 merge-readiness 下**通常仍算未完成** |
| **缺失** | 沒做 |
| **證據不足** | 看起來像有做，但無法證明（缺驗證證據） |
| **被反證** | 當前行為與要求矛盾 |
| **明確 descoped**（處置、非完成度狀態） | 經對齊決定不做、且**作者有留痕**（`02-plan.md` / issue / PR）—— 標出處才算數，無留痕的「沒做」一律歸 缺失。是「收斂去向」而非第六個完成度態（五態之一 + 此處置＝每條的兩種合法收斂） |

partial 要寫清楚：哪條路徑能 work、缺哪個狀態 / 入口 / 持久化。

> **完整性 gate（餵給 verify 步驟 4 acceptance 閘的出口判定）**：逐項 ledger **任一條停在 部分滿足 / 缺失 / 證據不足 / 被反證 且未明確 descoped → 判 Not ready**（partial 當完成是 P0/P1）。只有「每條都收斂到 已滿足（有證據）或 明確 descoped（留痕）」才放行。這道完整性檢查 **tier-independent**：凡 product-contract 有跑就生效（任何 issue，小孤立級〔3 軸〕起必派 product-contract），**不是只有高風險才做** —— 見 verify 步驟 4 acceptance 閘。

## 三、端到端鏈路 + 持久化驗證

若 issue / DoD 有 GWT 場景，端到端鏈路驗證以場景為單位：每條場景的 `Given` 佈置好、`When` 真的觸發、`Then` 在真實鏈路（使用者動作→傳輸→服務→持久化→回顯）可觀察到；無場景時照既有規則驗整條鏈（逐句 AC）。

- **只完成片段不算履約**：要驗整條鏈 `使用者動作 → 傳輸 → 服務 → 持久化 → 回顯` 都通；只證 UI 暫時變了、或只證後端有對應入口，都是 partial。
- **持久化要真**：凡 issue 要求保存 / 提交 / 編輯持久化，要驗**重新載入 / 重新查詢後結果仍在**；只有樂觀更新或快取更新**不算**持久化證據。
- 涉及非同步行為時，逐一對照狀態全集：開始 / 進行中 / 成功 / 失敗 / 空結果 / 重複觸發 / 取消 / 重入，是否都有對應實作。

## 四、驗證證據強弱分級

判斷 PR / 改動附的證據夠不夠強：

- **弱證據（不足以證明核心行為）**：「本地測過」沒附步驟、只貼一句 pass 看不出測了哪條、沒有前後操作的截圖、純從 code 推斷。
- **核心使用者行為**優先要至少一份**可執行**證據（測試 / 可重現步驟），不能只靠 code 推斷或弱證據。
- 跨端 / 跨層的行為，優先要求端到端可執行的驗證。

## 五、Finding 寫法

對齊雙視角：先工程視角（缺哪條 criterion / 哪條鏈斷在哪），再使用者視角（使用者會發現什麼沒做到）。partial / 證據不足 / 被反證都要明確標出，不含糊帶過。

## 六、驗收 checklist 產物（`CHECKLIST.md`，人類 / AI 共用、一律產）

acceptance 閘除了在報告裡逐條列五態，還要**把同一份逐項 ledger 物化成一份可勾選的 `CHECKLIST.md`**，寫到 `$LOOPS_ROOT/.loops/<slug>/CHECKLIST.md`。這份是**單一來源、兩種消費**：**AI 驗收**＝verify 跑真 app 時就地勾（把宣稱從 `not measured` 變實測）；**人類驗收**＝iterate 收尾交付、人照著手動玩一遍。**一律產、不吃任何旗標**（獨立於 `LOOPS_EXPLAIN`）；只有無驗收契約的瑣碎改動不產。

- **核對單位優先用 GWT 場景 ID（`S1…`，見 `bdd-scenarios.md`）**；無場景的 issue 退回逐句 AC。逐條就是逐項 ledger 的同一批（不另生一套、不漂移）。
- 每項＝**具體可執行步驟**（照著能操作，不是抽象描述）＋**可觀察預期**＋**AI 驗收欄**（三態 ＋ 證據錨點，Metric-Honesty）＋**人類勾選框**。
- AI 驗收三態：**✅ 通過**（附證據：`file:line` / 指令輸出 / 截圖路徑）｜**❌ 失敗**（連到對應 finding）｜**⬜ not-measured**（純 lib 無 app、或宿主無 `/run` 能跑，據實標，不假裝跑過）。

格式（範本）：

```markdown
# 驗收清單 — <issue 標題>（#<issue>）

> 人類手動驗收 ＝ AI 驗收共用。每項一條 GWT 場景 / AC 的具體步驟。
> 「AI 驗收」由 verify 跑真 app 標；左側 ☐ 留給人類手動勾。

## S1 永久刪除：擁有者刪自己的 trash item
- [ ] **操作**：以使用者 A 登入，對自己 trash 內的 item X 發 `DELETE /api/trash/X`
  - **預期**：回 204；重新 `GET /api/trash/X` 查不到（持久化，非樂觀更新）
  - **AI 驗收**：✅ 通過 — `curl` 得 204、重查 404（`trash.routes.ts:88`）

## S2 不可刪他人（授權邊界）
- [ ] **操作**：以 A 登入，對屬於 B 的 item Y 發 `DELETE /api/trash/Y`
  - **預期**：回 403/404；Y 仍在 B 的 trash（不被刪）
  - **AI 驗收**：❌ 失敗 — 回 204 且 Y 被刪 → 見 verify finding P0-1
```

> 這份 checklist **是 acceptance 閘的物化，不是另一道審查**：ledger 的每條收斂態直接決定 checklist 每項的 AI 驗收結果（已滿足→✅、被反證→❌、證據不足/純 lib→⬜）。與 `finding-validation.md`（驗「findings 是否真實/本次/已防護/對症」）正交、不重疊。
