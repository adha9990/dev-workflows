# Clean Architecture（設計 / 寫 code 的結構標準）

> `plan` 設計時的結構標準、`impl-author` 落實的方向、`architecture-reviewer` 的審查基準。管的是**模組 / 依賴 / 層級**的品質（line / 函式級見 `clean-code.md`）。

## 核心：依賴規則（The Dependency Rule）

依賴**向內流動**：外層（IO / 框架 / DB / UI）依賴內層（業務邏輯），**內層永遠不知道外層存在**。高層策略不依賴低層細節；雙方都依賴**抽象**，不依賴具體實作。

## 一、分層與邊界

- 每層**單一責任**：純邏輯（零 IO）→ 介面 → 編排 → 邊界（IO / 框架 / DB）。
- 跨層只透過**清楚的介面**，不繞過、不洩漏內部細節。
- 副作用與 IO **推到最外緣**，核心保持純淨、**不啟動基礎設施就能測**。

## 二、依賴反轉 + 注入

- 內層依賴**介面（port）**而非具體實作；具體實作（adapter）放在邊界。
- 凡是「**會碰外部世界**」的能力（DB / 外部 API / 檔案 / 時鐘 / 亂數）都值得一個介面。
- 用**工廠 / 建構式注入**接相依，**不在內層直接 `new` 基礎設施**；接線集中在 composition root。

## 三、內聚 > 耦合

- **一起改的放一起**（高內聚）；模組之間少而清楚的接點（低耦合）。
- 改動別讓某個檔 / 模組責任膨脹 —— 該拆就拆；加東西前先看它是不是已經太大。

## 四、落點對齊既有架構（screaming architecture）

- 新 code 對齊既有分層 / ports-adapters 慣例**放對位置**；**不憑空開新頂層資料夾**（該套既有典範卻另起爐灶）。
- 結構要「喊出它是什麼**領域**」，而不是「喊出用什麼**框架**」。

## 五、SOLID 精神（夠用即可，不教條）

單一職責、對擴充開放 / 對修改封閉、用介面隔離、依賴抽象 —— 當它們讓設計**更好懂 / 更好改**時才用，不為套而套。

## 六、YAGNI 與最高標準的界線

- **YAGNI 管 scope**：不做沒被需求要求的功能、不留投機抽象。
- **最高標準管 in-scope 品質**：要做的部分，對**可預見的規模 / 流量 / 並發退化**預先用對的結構與演算法（見 `plan` §4 六維度）。
- 兩者不衝突：YAGNI 砍掉「不做的」，最高標準把「要做的」做對。

## 紅旗

- 內層 import 外層（核心邏輯依賴 DB / 框架 / UI）；依賴成環；反向依賴。
- 業務邏輯與 IO / 框架糾纏，**不啟動基礎設施就測不了核心**。
- 在內層直接 `new` 具體基礎設施（沒走注入）。
- 憑空開頂層資料夾、不對齊既有典範。
- 為臆測（非需求）預留一堆抽象層。

## Verification

- [ ] 依賴向內：內層不 import 外層、無環、無反向依賴。
- [ ] 副作用 / IO 在邊界，核心純淨、可獨立測試。
- [ ] 外部能力走 port + adapter，相依用注入、不在內層 `new`。
- [ ] 新 code 對齊既有分層 / 落點，沒憑空開頂層資料夾。
- [ ] in-scope 品質照最高標準（規模 / 並發），但沒做 YAGNI 的投機抽象。

## Domain-Driven 詞彙（右尺寸，只在碰領域概念時）

- **Ubiquitous Language**：同一個領域概念，在 issue / DoD 場景 / glossary / code identifier / PR comment **用同一個名詞**。命名漂移＝缺陷（architecture-reviewer 會抓）。
- **Entity / Value Object / Aggregate**：設計動到領域物件時顯式分類——Entity（有 ID、可變、生命週期）/ Value Object（無 ID、不可變、以值相等）/ Aggregate（一致性邊界 + root，外部只透過 root 操作、不變式在邊界內維持）。
- **Bounded Context（BC）**：同一名詞在不同脈絡可能意義不同；跨 BC 邊界要明確（探索時辨識，見 explore）。
- **右尺寸**：瑣碎 / 純 refactor / 不碰領域的改動**跳過**這些；新功能命名動到的物件即可，高風險 / 核心領域才完整建模。不對小任務加 ceremony（規則 10）。
