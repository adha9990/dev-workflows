# Clean Code（寫 code 的正向品質標準）

> `impl-author` 寫實作時照著寫的標準、`code-quality-reviewer` 的審查基準。核心：**一次就寫乾淨**，不是先寫爛再靠 refactor 救（refactor 規範見 `code-simplification.md`、重複收斂見 `reuse-check.md`、測試品質見 `test-rubric.md`）。

## 核心：code 是寫給人讀的

一段 code 被讀的次數遠多於被寫。寫的當下就讓「下一個讀的人（包含三個月後的你）一眼看懂」。標準不是「能跑」，是「**好讀、好懂、好改、好 debug**」。

## 一、命名揭示意圖

- 名字說明「**為什麼存在、做什麼**」，不是「怎麼實作」。`elapsedDays` 勝過 `d`。
- 避免縮寫、魔法、誤導；**同一個概念全程用同一個詞**（不要 user / account / member 混用同一件事）。
- 好名字能**消滅註解** —— 名字夠清楚，解釋「what」的註解就多餘。

## 二、函式小而單一職責

- 一個函式**做一件事**、停在**同一個抽象層級**（別在一個函式裡又調框架又算商業規則）。
- **guard clause / early return** 取代深層巢狀：先把邊界、錯誤、前置條件擋掉再走 happy path。
- 參數宜少；> 3 個考慮包成物件。**布林參數**常是「該拆成兩個函式」的訊號。

## 三、錯誤處理顯式、不吞

- 不空 catch、不把錯誤吞成預設值、不讓失敗看起來像成功（呼應 verify 的 silent-failure 軸）。
- 錯誤**型別化**、帶足以診斷的 context；邊界（null / 空 / 溢位 / 逾時 / 併發）明確處理，不留「不會發生啦」。

## 四、用型別表達契約

- 不 `any`、不無謂強轉；nullable **顯式**標出。
- **讓非法狀態無法被表示**：用型別 / 列舉 / 精煉型別把不變式編進型別，而不是靠執行期注釋或文件約定。

## 五、常數具名、無魔法值

- 魔法數字 / 字串抽成**具名常數**，名字解釋意義（`const MAX_RETRY = 3`），別讓 `3` 裸奔在邏輯裡。

## 六、註解講 why 不講 what

- 解釋「**為什麼**」（意圖、取捨、外部限制、踩過的坑）的註解保留；複述 code 在做什麼的刪掉（細節見 `code-simplification.md`）。
- **來源追溯歸 commit / PR，不進 code 註解**：issue / PR 編號、「當時怎麼壞→怎麼修」的情境敘事寫進 commit message / PR body（`git blame` 一步可達），不複寫進 code。「踩過的坑」留的是**坑的約束**，不是坑的故事：
  - ✗ `// Issue #NNN：當時 tooltip 會在關閉後彈回，因為…（20 行修 bug 史 + review 出處）`
  - ✓ `// 套件在 outside 關閉時不做 focus-return，flag 需在此解除，否則吃掉下一次合法 hover`
- **密度對齊鄰近同類檔**：同專案同類元件的註解佔比是基準，單檔明顯高出（如 16% vs 鄰檔 7–8%）＝敘事過多的訊號；每處 why 以 2–5 行為度。

## 七、重複：產品碼適度 DRY、測試碼 DAMP

- 產品碼避免**真重複**（同一件事散在多處）；但**稍異 ≠ 強合** —— 入口稍異優先參數化既有方法，不另造（判準見 `reuse-check.md`）。
- 測試碼 **DAMP > DRY**：每個測試自我說明、別為去重犧牲可讀（見 `test-rubric.md`）。

## 紅旗

- 打算「先能跑、之後再 refactor 救」 —— 寫的當下就該乾淨，refactor 是精修不是補救。
- 函式長到要捲動才看完 / 巢狀 > 3 層 / 一個函式跨好幾個抽象層級。
- `any`、空 catch、裸魔法值、揭示不了意圖的命名。
- 為「之後可能會用」先放抽象（投機抽象＝沒價值的複雜度，要時再加）。

## Verification

- [ ] 函式單一職責、同一抽象層級，邊界 / 錯誤用 guard clause 先處理。
- [ ] 無 `any` / 空 catch / silent failure；錯誤型別化、帶 context。
- [ ] 無裸魔法值；註解只剩 why，無 issue/PR 編號與情境敘事，密度不高於鄰近同類檔。
