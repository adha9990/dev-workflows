# Design Patterns（設計模式：對症的可複用解法）

> `plan` 設計時的結構選型參考、`architecture-reviewer` 的審查基準、`refactoring.md` 重構引入模式時的目錄。**核心紀律：模式是用來解決「反覆出現的設計問題」的，不是用來炫技 —— 對症才用、不為套而套**（YAGNI / 過度設計是反例，呼應 `clean-architecture.md` 的 SOLID「夠用即可、不教條」）。

## 兩個時機

- **設計時（`plan`）**：設計某機制時，若問題本來就是某模式的**經典形狀**（多變體、要可替換演算法、要解耦通知…），直接用對的模式比之後重構便宜。
- **重構時（build Refactor）**：某種異味**反覆出現**（條件巨獸、Duplicated Code、緊耦合）→ 引入模式消除它（見 `refactoring.md`）。

## GoF 三大類 + 何時對症

### 建立型（物件怎麼生）
- **Factory Method / Abstract Factory**：建立邏輯多變體、要與使用端解耦。
- **Builder**：組裝步驟多 / 參數爆炸（呼應 Long Parameter List）。
- **Singleton**：謹慎 —— 常淪為全域狀態（耦合 / 難測），多數時候**依賴注入**更好。

### 結構型（物件怎麼組）
- **Adapter**：接既有介面不合的東西（包一層轉換）。
- **Decorator**：動態加行為、不想爆繼承組合。
- **Facade**：給複雜子系統一個簡單入口（呼應 clean-architecture 邊界）。
- **Composite**：樹狀 / 整體-部分一致對待。
- **Proxy**：延遲載入 / 存取控制 / 快取包裝。

### 行為型（物件怎麼互動）
- **Strategy**：同一件事多種**可替換演算法**（取代「選演算法」的條件分支）。
- **State**：行為隨狀態變（取代散落的狀態 if / switch）。
- **Observer**：一對多通知 / 事件訂閱、要解耦發布者與訂閱者。
- **Template Method**：流程骨架固定、步驟可變（取代重複的流程骨架）。
- **Command**：把操作物件化（佇列 / undo / 記錄）。

## 反面：別過度設計（pattern 上癮）

- 不為套而套：簡單 if / else 換成五個類**不是**進步。
- 模式要讓設計**更好懂 / 好改 / 好測**才用；做不到就是徒增複雜度。
- **YAGNI**：沒有反覆出現的問題，就別預先套模式（投機抽象）。

## 紅旗

- 先選模式、再硬把問題塞進去（而非問題本來就是那個形狀）。
- 一個小功能套了三個模式。
- 用 Singleton 當「全域狀態」的藉口。
- 模式名掛滿、卻沒減少任何實際複雜度（複雜度只是搬家）。

## Verification

- [ ] 引入的模式對症一個**真實、反覆出現**的設計問題（不是炫技 / 投機）。
- [ ] 模式讓設計更好懂 / 好改 / 好測，而非更繞。
- [ ] 沒有為了套模式把簡單邏輯複雜化。
- [ ] Singleton（若用）不是在掩蓋全域狀態 —— 優先依賴注入。
