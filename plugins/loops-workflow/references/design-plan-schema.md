# design-plan-schema —— `stages/02-plan.md` 設計計畫文件的完整施工圖骨架

> plan 產 `stages/02-plan.md` 時的完整施工圖骨架（§0–§9）。

`stages/02-plan.md` 不是「修改清單」，是一份**施工圖**：讓工程師在**動 code 之前**就看懂整個系統**怎麼運作 + 為什麼這樣設計 + 誰背書 + 做完長什麼樣**。下面的 §0–§9 是這份文件的固定骨架（順序固定）。plan 的 `Process` 各步驟產出的內容，**全部歸位**到這套骨架裡 —— 這份 schema 不取代 plan，而是把 plan 已有的零件組織成一份完整文件，並補上 plan 沒寫明的缺口（系統全貌、檔案職責表、名詞說明、三角驗證、每條決策的具名背書）。

## 核心精神

- **施工圖視角，不是修改紀錄**：正面寫「每個檔在系統裡負責什麼」，**不用**「本次新增 / 改了什麼」「🆕NEW／✏️MOD」這種相對現況的 delta 措辭。讀者要看「系統長怎樣」、不是考古「你改了什麼」（迭代脈絡靠 git history）。
- **每條決策四件齊全**：`選擇 / 為什麼 / 背書 / 未採用` —— 每個方向都要有具名大型開源專案或硬需求撐著，不自己憑空設計；背書欄絕不可空（只有單一非大型來源就寫「僅 X 參考、屬選項」）。
- **白話優先 + 圖輔助**：名詞、全貌、決策盡量配生活比喻 + mermaid，讓非該領域的人也讀得懂。
- **living source of truth**：實作期偏離就回來更新（保持 as-built）；完工時這份文件提煉成 PR body。

## §0–§9 骨架（每段一句話「放什麼」）

| § | Section | 放什麼（一句話） |
|---|---|---|
| 0 | 標題 + 一句話 | 標題（feature + issue 號）+ 出處 + 一句話這次在做什麼 —— 開門見山。 |
| 1 | 系統運作全貌 + 系統架構圖 | 端到端運作流程圖（觸發→處理→輸出）+ 分層架構圖（標新元件落哪層）+ 資料流／注入流施工圖 + **檔案落點與職責表**（每個檔：層・職責・對齊哪個既有典範）。 |
| 2 | 機制與功能圖文說明 | 每個關鍵機制一小節，「**一段白話 + 兩張圖**」齊全：運作流程圖 + 注入／接線圖（節點標到檔案層級）—— 只有文字、沒有圖不算數。 |
| 3 | 名詞說明 | 白話 glossary（什麼是 X / 什麼是 Y），讓非該領域的人讀得懂後面的決策。 |
| 4 | 設計目標與原則 | 北極星原則 + 驗收試金石；以**設計品質六維度**（簡潔／可維護／可靠／可擴展／安全／高併發高流量效能）為北極星，寫明對可預見規模退化的最高標準做法。 |
| 5 | 參考來源 | 研究過的大型開源專案 + 各自份量／管哪些決策 —— 證明決策有實證背書、不是憑空。 |
| 6 | 核心決策 | 套件清單表（若有新套件）+ 每條決策 **選擇 / 為什麼 / 背書（具名專案+怎麼做）/ 未採用 / 拍板人** —— 文件主體、audit trail 核心。 |
| 7 | 不在本次範圍 | 刻意延到後續 / 等外部輸入才能拍板的事項 —— 框清這次邊界。 |
| 8 | 附錄：三角驗證 | 表：每個分歧 × 各來源各做什麼 + 我們的選擇 —— 一眼看出每個選擇有幾票支持。 |
| 9 | 預計成果展示 →（完工後）成果展示 | **計畫版**：實作後預計輸出長什麼樣（範例輸出 + 對應哪條決策 + 怎麼查／用）；**as-built 版**：改名「成果展示」，填真跑一輪的真實行為 + 效能數字。 |

## plan 既有零件如何歸位到 §0–§9

這份 schema **superset** 了 plan `Process` 已有的步驟，不與它衝突。對照表：

| plan/SKILL.md 的步驟 | 落在 §0–§9 哪一段 |
|---|---|
| 步驟 1 決策留痕（decision record 五欄） | **§6 核心決策**（五欄決策 + 拍板人） |
| 步驟 2 套件評估（≥3 候選比較） | **§6** 開頭的套件清單表 + 每套件決策 |
| 步驟 3 機制圖（每機制 白話 + 兩圖） | **§2 機制與功能圖文說明** |
| 步驟 4 品質維度 + 重用檢查 | **§4 設計目標與原則** |
| 步驟 5 拆可驗證任務 | 任務清單（在 `stages/02-plan.md` 內、§0–§9 文件本體**之後**；schema 不重述，見 `task-template.md`） |
| 步驟 6 送出 + 拍板 gate 渲染機制圖 | gate 渲染的就是 **§2** 的兩張圖（schema 不改 gate 行為） |

**plan 沒寫明、本 schema 補上的缺口**：§1 系統全貌 + **檔案落點與職責表**、§3 名詞說明、§5 參考來源、§6 每條決策的**具名 OSS 背書**欄、§8 三角驗證附錄、§9 成果展示。這些是「機制圖 + 任務清單」被掏空後失落的施工圖骨幹 —— 補回來，`stages/02-plan.md` 才是完整施工圖而非只有兩塊。

## 各段細部規格

### §0 標題 + 一句話
`## <feature> 設計決策（#<N>）` + workflow 出處 + 一句話講這次在做什麼。

### §1 系統運作全貌 + 系統架構圖
**兩類圖 + 一張表**：
- ① **運作流程圖**（端到端 mermaid：觸發→處理→輸出→查詢）+ 白話。
- ② **系統架構圖** ＝ 分層概念圖（對齊既有分層、標新元件落哪層）**＋資料流／注入流施工圖**（展開「那個東西怎麼從進入點流到每個 service、各檔負責什麼」；凡有 composition root／DI／組裝層的功能必畫）。
- ③ **檔案落點與職責表**（每個檔：層・職責・對齊哪個既有典範）。
- ④ **跨介面契約**（API / 資料模型 / 事件動到時）：對外承諾的形狀 —— request / response / 錯誤、schema + 約束 + migration 可逆性、事件 payload + 保證，逐條對到測試層。詳見 `contract-spec.md`。

**必須對齊既有架構**：落點依據是對目標模組關鍵檔的實讀（分層／ports-adapters 慣例／import 規則），套既有典範決定落點並寫明「為什麼這樣放」。**嚴禁憑空開新頂層資料夾**。命名表與異動表高度重疊時合併成一張完整職責表。

### §2 機制與功能圖文說明
**每個機制 / 功能一小節，「一段白話 ＋ 兩張圖」齊全**：
- ① 白話一段（解決什麼問題）。
- ② **運作流程圖**（mermaid：觸發→判斷→輸出，含關鍵分支與失敗路徑）。
- ③ **注入接線圖**（mermaid：進入點／composition root → deps 流向 → 掛載點，節點標到檔案層級，與 §1 施工圖對得上；圖後配一兩句補節點放不下的細節）。

**注入說明只有文字不算數，必須有圖**。機制清單以**實作完成時**為準 —— 實作期新增的機制必須回寫補進本段（活文件）。這正是 plan 步驟 6 拍板 gate 要直接渲染給使用者看的兩張圖。

### §3 名詞說明
白話 glossary（什麼是 X / 什麼是 Y），讓非該領域的人讀得懂後面的決策。

§3 是 **Ubiquitous Language glossary**：白話定義每個領域名詞，並（碰領域時）標 **〔Entity〕/〔VO〕/〔Aggregate〕**。這些名詞是 living 的單一真相源——issue / DoD 場景 / code identifier / PR 一律沿用同名（不一致由 architecture-reviewer 抓）。右尺寸：不碰領域的改動 §3 可從簡。

### §4 設計目標與原則
北極星原則 + 驗收試金石。以**設計品質六維度**（簡潔／可維護／可靠／可擴展／安全／高併發高流量效能）為品質北極星，寫明本 feature 對可預見規模退化（資料量／流量成長）的最高標準做法 —— in-scope 不以 MVP 設計。重用優先的結論（為何不重複造輪子）也在此交代。

### §5 參考來源
研究過的大型開源專案 + 各自份量／管哪些決策。證明決策有實證背書、不是憑空。**外部參考只有參考價值** —— 寫成「佐證 + 推薦」，不可寫成「採用 / 已決定」（決策權在使用者）。

### §6 核心決策
套件清單表（若有新套件）+ 每條決策走五欄：

- **選擇**：拍板選項。
- **為什麼**：為什麼選這個，點出本次的硬限制。
- **背書**：具名大型開源專案 + 它怎麼做；**絕不可空**（只有單一非大型來源就寫「僅 X 參考、屬選項」，不假裝有共識）。
- **未採用的選項**：其他選項 + 為什麼不選（trade-off 的 audit 紀錄）。
- **拍板人**：對齊本機 `git config user.name`；Claude 自決才標 `Claude（依據 <convention>）`。

套件清單置 §6 開頭：**本次必裝表**純列 id／類型／用途；**每個套件決策**走同一套五欄。只列本次必裝，沒新套件就整段拿掉。

### §7 不在本次範圍
只放 (a) 明確排除／拆去未來的功能、(b) 等外部輸入才能拍板的事項。**不要**放「實作期邊做邊量測才定的細節」（那是猜不是決策），**也不要**放「驗收約束 / invariant」（那是要做要守的事）。判準：問「這需要某人之後拍板、或推到未來嗎？」答是才留。

### §8 附錄：三角驗證
一張表：每個分歧（決策點）× 各參考來源各做什麼 + 我們的選擇。讓讀者一眼看出每個選擇有幾票支持、是不是孤證。

### §9 預計成果展示 →（完工後）成果展示
- **計畫版**：實作後預計輸出長什麼樣（範例輸出 + 逐條標「對得上哪條決策」+ 怎麼查／怎麼用）。
- **as-built 版**：完工提煉成 PR body 時改名「成果展示」，填真跑一輪的真實行為 + 效能數字（不是預期值）。

**對 output-type 功能必備**：logger / API / formatter / CLI 這類「成果就是輸出」的，§9 必放真實範例輸出（逐條標對得上哪條決策）；純內部重構改放「行為對照（前 → 後）」。

## 通用規則

- **去無謂反面詞**：凡「我們不做 X」而 X 本來就不是預設、讀者不會預期，刪掉。反面聲明只在反駁「讀者看不到的外部主張」時才有意義（§6 決策的「未採用」欄例外 —— 那是 trade-off 紀錄）。
- **§1 必加施工圖**：只畫分層概念圖、沒畫資料流／注入流施工圖 = 看得到「誰能依賴誰」卻看不到「東西怎麼從進入點流到每個 service」；凡有 composition root／DI／組裝層的功能，§1 必加注入流施工圖 ＋ 檔案落點與職責表。
- **§2 每機制兩圖不可省**：只有文字沒有圖 = 每個讀者腦內自己畫運作模型、各畫各的；注入只有文字 = 「誰建構誰、deps 流向哪、掛在哪」得逐句重組。
- **as-built 紀律**：完工時 §9 改名「成果展示」填真實結果，其餘段落回寫成實況；**不**新增設計更新 / commit / 後續迭代記錄段 —— 文件描述「系統長怎樣 + 真實成果」，不是修改流水帳。
