# 文件與開發者體驗審查（docs-devex-review）

> docs-devex-reviewer 用（條件式）：不是文案潤稿。只抓**影響交付 / 驗證 / 操作 / 維護**的文件缺口，以及 PR 自身作為一份操作文件夠不夠用。

## 一、文件同步審查（這次改動讓既有文件變誤導沒）

> 這是**審查視角**（下一個人照現有文件操作會不會出事），與「作者該不該補新文件」（見 `docs-policy.md`，產出視角）不同。

- **公開流程 / 介面改了，既有文件仍指向舊做法** → 照舊文件操作會造成錯誤 / 不可恢復狀態（P1）。
- 範例命令 / 範例 payload / 路徑 / 設定鍵改了，文件裡的對應內容是否還對。
- 對外 API / 契約 / 錯誤形狀 / 事件 payload 改了，對應的 reference 文件 / 型別說明是否同步。
- 設定 / migration / 升級步驟改了，有沒有對應的 upgrade / migration note。
- 專案專屬命令（產生程式碼、建置、初始化等）的用法變了，對應指引是否同步。
- **docs 宣稱 vs 實作交叉驗證**：對**改動觸及的子系統**，把該區 `docs/<topic>.md` 裡的具體宣稱（範例 payload / 檔案格式 / 回傳形狀 / 欄位）**逐一對照實際 code 行為**，不符即報——**即使該不符早於本次改動**（例：doc 寫「匯出下載的就是 raw overrides」，實際是含 `version`/`type`/`exportedAt` 的 envelope）。改動 IN 該子系統時，這份 doc 就是接手者會照著做的權威；照它做會失敗 = P1/P2。（與上面「這次改動讓既有文件變誤導」互補：這條抓的是「文件本來就與實作不符、而本次正好碰到該區」。）
- **行為變更 → 全 `docs/` 關鍵字掃（別只查改動檔附近的 doc）**：當本次改動改的是一個**行為 / 契約**（不只某個檔案），doc 漂移要照「**改了哪個行為**」找、不是照「改了哪些檔」找——描述舊行為的文件常散在**離改動 code 很遠**的 topic doc 裡。作法：拿「被改掉的**舊行為**關鍵字」（例：某模型扁平化後的 `recursive X` / `X tree` / `parent_id`；某 API 改形狀後的舊欄位名 / 舊端點）**grep 整個 `docs/`（＋所有 `AGENTS.md`/`CLAUDE.md`）**，逐一核每個命中是否還在教舊行為。只更新「改動檔旁邊那幾份 doc」是最常見的漏抓。（實例：#219 tag 扁平化，貼著 code 的 `smart-folder/AGENTS.md` 有改，但 `docs/services/smart-folder-query-compiler.md`、`item-query.md`、`agent-doc-coverage.md` 三份 topic doc 仍描述舊的 recursive tag tree、漏抓，拖到人工 reviewer 才發現。）

## 二、PR body 驗證證據品質

PR body 是 reviewer 拿到的第一份操作文件：

- 有沒有講清楚**改了什麼 / 為什麼 / 怎麼驗**。
- 有沒有列出**沒驗到的範圍**（誠實標未覆蓋，比假裝全測過好）。
- **「本地測過」沒有可重現步驟 = 弱證據**，不能當強驗證（呼應 Metric-Honesty：沒實跑的宣稱標 `not measured`）。
- 聲稱「不需要改文件」時，對照第一節確認真的沒有公開介面 / 流程 / 操作方式變動。

## 三、觸發與範圍

- 觸發：動到 docs / README / 模組說明檔，或 CLI / setup / migration / config / 對外 API / 錯誤形狀 / 自動產生型別，或 PR body 聲稱免改文件。
- **不觸發**：純內部重構、小型 bug fix，沒改任何對外操作方式 —— 不要為了湊而要求補文件。

## 四、docs 自足教學品質（本次新增 / 改動了 `docs/<topic>.md` 時**必查**）

`docs/<topic>.md` 是**獨立教學文檔**，不是交付紀錄。對本次改動新增 / 動到的每一份 `docs/`，逐項對照〈`docs-policy.md` → docs 必須自足〉，違反即報：

- **引用了 issue / PR 號**（`#NNN`、「見 issue #X」「承 #Y 研究報告」「設計脈絡見 PR #Z」，或 `.loops/` 路徑）→ **P2 finding**：教學文檔該自足，讀者不該被迫翻票才懂；改成直接寫進 doc，要指更細只指其他 `docs/`、`AGENTS.md`、`file:line`。
- **含狀態 / roadmap / 交付進度段**（「現狀與後續」「Phase 1/2 已交付」「後續 follow-up」「里程碑」「已交付/尚未做」清單）→ **P2**：那是 PR body / plan 的事；doc 只寫「現在怎麼運作 + 限制」。
- **把限制寫成進度**（「Phase 2 才做 X」而非「目前不支援 X」）→ 報，建議改寫成限制陳述。
- 「為什麼這樣設計 / 取捨理由 / 被否決方案」混進 docs（該在 PR body / plan）→ 報。
- **不白話 / 寫給已懂的人**：術語第一次出現沒就地解釋（`t()`/`catalog`/`schema-driven`/`resolvedLanguage`… 當讀者已懂）、一開頭就丟架構表 / 通篇縮寫、沒有心智模型或「跟著走一遍」→ 報：doc 要能讓**不懂、剛進來**的人從上到下讀懂，不必先會（見 `docs-policy.md`〈寫給不懂、剛進來的人〉）。
- **圖用 ASCII art 而非 mermaid**（手畫箭頭 / 方框圖）→ 報：改成 ` ```mermaid ` flowchart（GitHub 原生渲染、好維護、螢幕閱讀器友善）。
- **缺〈怎麼自己驗證〉段**（只描述系統怎麼動、沒給讀者親手確認的方法：打哪個 API 看什麼欄位 / 查哪張表 / DevTools 看什麼 / 重載看什麼）→ 報：教學要能讓讀者驗證，不能只描述。

（這節是**產出品質**視角，與第一節「文件同步」的**誤導**視角互補：一份 doc 可以與實作完全同步，卻因引用了票號 / 塞了交付進度 / 不白話 / 沒圖沒驗證而不合格。）

## 五、Finding 寫法

雙視角：工程視角（哪份文件 / 哪段 PR body、哪裡與現況不符），使用者 / 接手者視角（下一個人照它操作會踩到什麼）。只報會實際誤導人或卡住驗證 / 操作的、或違反 docs 自足教學品質的，不報純排版 / 措辭偏好。
