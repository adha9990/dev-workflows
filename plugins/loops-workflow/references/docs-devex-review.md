# 文件與開發者體驗審查（docs-devex-review）

> docs-devex-reviewer 用（條件式）：不是文案潤稿。只抓**影響交付 / 驗證 / 操作 / 維護**的文件缺口，以及 PR 自身作為一份操作文件夠不夠用。

## 一、文件同步審查（這次改動讓既有文件變誤導沒）

> 這是**審查視角**（下一個人照現有文件操作會不會出事），與「作者該不該補新文件」（見 `docs-policy.md`，產出視角）不同。

- **公開流程 / 介面改了，既有文件仍指向舊做法** → 照舊文件操作會造成錯誤 / 不可恢復狀態（P1）。
- 範例命令 / 範例 payload / 路徑 / 設定鍵改了，文件裡的對應內容是否還對。
- 對外 API / 契約 / 錯誤形狀 / 事件 payload 改了，對應的 reference 文件 / 型別說明是否同步。
- 設定 / migration / 升級步驟改了，有沒有對應的 upgrade / migration note。
- 專案專屬命令（產生程式碼、建置、初始化等）的用法變了，對應指引是否同步。

## 二、PR body 驗證證據品質

PR body 是 reviewer 拿到的第一份操作文件：

- 有沒有講清楚**改了什麼 / 為什麼 / 怎麼驗**。
- 有沒有列出**沒驗到的範圍**（誠實標未覆蓋，比假裝全測過好）。
- **「本地測過」沒有可重現步驟 = 弱證據**，不能當強驗證（呼應 Metric-Honesty：沒實跑的宣稱標 `not measured`）。
- 聲稱「不需要改文件」時，對照第一節確認真的沒有公開介面 / 流程 / 操作方式變動。

## 三、觸發與範圍

- 觸發：動到 docs / README / 模組說明檔，或 CLI / setup / migration / config / 對外 API / 錯誤形狀 / 自動產生型別，或 PR body 聲稱免改文件。
- **不觸發**：純內部重構、小型 bug fix，沒改任何對外操作方式 —— 不要為了湊而要求補文件。

## 四、Finding 寫法

雙視角：工程視角（哪份文件 / 哪段 PR body、哪裡與現況不符），使用者 / 接手者視角（下一個人照它操作會踩到什麼）。只報會實際誤導人或卡住驗證 / 操作的，不報純排版 / 措辭偏好。
