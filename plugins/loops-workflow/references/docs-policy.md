# docs 規範（什麼該寫、什麼別寫、何時更新規範檔）

> 目的：讓「後續維護者看得懂」，同時避免「隨便小功能就塞 docs/」。在 build 完成 / iterate 收尾時，把「該不該補 docs」當成完工定義的一部分。

## 何時寫 `docs/<topic>.md`

命中**任一**就寫：
- **新子系統**（logging / i18n / auth / 權限 / 快取…）。
- **跨切面關注點**（影響多模組、不屬單一檔案）。
- **未來維護者光看 code 看不懂的設計決策 / 約定**（為什麼這樣切、有什麼坑）。

## 何時不寫

- 單一小功能、看 code 就懂的。
- 一次性 / 即將被取代的。
- 只是 API 列表（那交給型別 / 簽章，不要 docs dump）。

## docs 寫什麼

**教學取向：怎麼用、有什麼限制 / 坑、怎麼擴充。** docs = 教學手冊（how-to），**不是決策紀錄** —— 「**為什麼這樣設計 / 取捨理由 / 被否決的方案**」屬於設計計畫（`design-plan-schema.md` §6 決策 / §8 三角驗證）與 PR body，不重複塞進 docs（docs 只在「不寫使用者會踩坑」時點到設計約定，不展開決策 audit）。不重貼 code、不寫成 API 字典。

## 維護 `docs/README.md` 索引

每新增一個 `docs/<topic>.md`，就在 `docs/README.md` 補一行（標題 + 一句話）。沒有索引就一行：`- [logging](logging.md) — 日誌系統怎麼用 / 怎麼設計`。讓人從索引就能找到所有主題文件。

## 何時更新 `AGENTS.md` / `CLAUDE.md`

**只在「慣例 / 規則本身改變」時**，不是每個 feature：
- 新增一條團隊規範 / 開發約定。
- 改了分層約定、目錄結構約定。
- 新的命名 / 錯誤處理 / 測試慣例。

純功能改動**不動**這兩個檔。

## 判斷流程

```
build 完成 / iterate 收尾
  ├─ 新子系統 / 跨切面 / 設計不直觀？ → 寫 docs/<topic>.md（+ 更新 docs/README.md 索引）
  ├─ 改了慣例 / 規則？             → 更新 AGENTS.md / CLAUDE.md
  └─ 單一小功能 · 看 code 就懂？     → 不寫（避免 docs/ 膨脹）
```
