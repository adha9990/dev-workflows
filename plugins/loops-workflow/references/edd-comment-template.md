# EDD comment 版型（研究／提案對外交付）

> 研究票（deep research / 架構選型 / 工程提案）的最終交付 post 成 issue comment 時固定此版型（`outbound-templates.md` 第 7 型）。讀者是要拍板「這個方向要不要做」的人——整份文件要讓人不讀原始研究也能審方案、看懂取捨、知道下一步怎麼拆。
>
> 區塊是**固定順序的骨架**、依題目右尺寸：題目沒有對應面的區塊可併入鄰近區塊（要在文內講明、不默默跳過）；小題不硬湊九區塊。

## 適用時機（先辨文體再選版型）

- `explore` / `clarify` 產出研究報告的**決策摘要**（研究細節可分多則 comment，EDD 是收斂拍板的那份）。
- design 型 loop 的**工程提案**交付。
- `iterate` 回應審查意見的**提案修訂版**（vN）。

**與其他版型的分界（辨文體、不是辨階段——同一個 iterate 階段可能發 §8 也可能發 EDD）**：

| 文體 | 版型 |
|---|---|
| 提案／研究——還沒做、要人拍板「做不做、怎麼做」 | 本檔 EDD |
| 驗收——審查做完的東西能不能合併 | `comment-policy.md` §7 |
| 修正回覆——自己修完 PR 回饋回報「修了什麼」 | `comment-policy.md` §8 |
| 拍板後的實作對齊（設計書 as-built） | `plan-comment-template.md`（位於 `skills/plan/references/`）|

## Header 與開頭定位

- 第一行：`## 📐 Engineering Design Document — <題目>`；修訂版加註（例 `（修訂版 v2）— <題目>`）；亦可視需要加描述性副標（例 `— <題目>（研究摘要與推薦方案）`）。
- 緊接**blockquote 文件定位**，固定講清楚：
  1. 本文件是什麼（決策摘要版／提案修訂版…）＋**引用完整研究來源**（前置研究 comment 的 GitHub 連結；細節論證在那些 comment、本文不重複）。
  2. **「本提案經審核同意前不進入任何實作」**；研究票再加「不承諾實作範圍或排期」。
  3. 修訂版另講：回應哪則審查意見（連結）、核心改動幾點、取代原版哪些節。

## 固定區塊（依序）

1. **研究摘要**——先講「怎麼研究的」再列主要結論。證據紀律：原始碼／文件**確認**的結論與**推論**分開標；有獨立覆核就寫覆核結果（幾條通過／駁回）；結尾一句「對本專案的含義」。
2. **推薦方案**——**一句話版**起頭（10 秒抓到方案形狀）再展開機制；放 **mermaid 機制圖**（GitHub 原生渲染）；含「與既有基建的接線」（每個整合點指既有先例／`file:line`）。
3. **領域清單表**——題目的核心枚舉（操作清單／欄位型別清單／端點清單…依題命名），逐項含優先序＋理由；閾值等未經實測的數字明標「工程判斷、非實測」。
4. **取捨分析（推薦與不推薦方案）**——為什麼是推薦案而不是各替代案（逐案講）；**「便宜但留債」的方案必須明標它埋的債**（AGENTS 規則 10）；「第一階段刻意不做」的列出＋理由。
5. **架構方向**——方案的架構形狀一句話＋具名前例；**誠實聲明：組裝 vs 照抄**（業界無單一可照抄前例就明講「這是組裝、每個零件各有出處」）。若本節比較實作位置／架構形狀的替代選項，同第 4 區塊紀律逐案標明「便宜但留債」。
6. **入口一致性**——多入口／多消費者怎麼自動涵蓋、怎麼防繞過、不出現分裂實作。題目無多入口面 → 併入〈架構方向〉並講明。
7. **風險與限制**——每條風險附對策；規模／效能主張寫成**可否證假設**（H1、H2…，講清楚用什麼實測驗證、不過關退什麼）；沒實測的標 `not measured`；**待人拍板的決策**單獨列出、不埋進內文。
8. **後續實作拆票（審核通過後）**——表：`# / 票 / 內容 / 相依`；地基票與可平行票分清楚（薄地基 → 寬平行）。
9. **驗收標準對照**——表：issue 驗收標準 ↔ 交付位置（本文哪節／哪則 comment），逐條對上、不漏條。

結尾：`> loops-workflow <stage> 留痕`（stage＝實際發文階段：explore / iterate…）。

## 修訂慣例（與 living 版型不同）

提案被審查退回後的修訂**發 vN 新 comment**（blockquote 聲明回應誰、取代哪些節），**不原地 PATCH 舊版**——審查意見與各版提案是時間軸上的對話，原地改寫會讓已發佈的審查意見失去對象。這與 plan／verify／iterate comment 的 living（as-built 偏離回去 PATCH）刻意不同。

## House-style（同全型）

套 `comment-policy.md`：§0 self-contained 不引 `.loops/` 路徑、§1 繁中、§2 白話（研究行話留在研究 comment，EDD 收斂處講人話）、§5 tmp 草稿 → 使用者確認 → 送出 → 刪、§6 不客套。
