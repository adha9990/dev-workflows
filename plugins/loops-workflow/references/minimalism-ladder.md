# minimalism-ladder — 動手前的最小主義階梯（單一正本）

> impl-author 動手加任何新東西（方法 / 檔案 / 抽象 / 欄位 / 依賴）前，**依序爬這條階梯**，停在最早能滿足需求的那一階——author-time 就砍掉不必要的，不留給 verify 抓。核心紀律：**能不做就不做、能用既有的就不新造、能少寫就少寫。**

1. **這東西真的需要做嗎？** 你要加的每一項，對應得到 `00-goal.md` 哪一條需求 / 驗收？答不出就別加。（**逐動作問**：這不同於 `clean-architecture.md` YAGNI 的 scope 層——那管「整體功能要不要做」、一次性判斷；這裡是每個新方法 / 檔案 / 欄位都單獨答。）
2. **codebase 已經有能複用的嗎？** 見 `reuse-check.md`（同方法不同入口、跨 session / 跨層的隱蔽重複）。
3. **標準庫 / 語言內建能做嗎？** 別為一個小工具引依賴——先看語言內建（陣列 / 字串 / 日期 / crypto…）。
4. **專案框架的原生能力能覆蓋嗎？** 專案實際用的框架有沒有內建能力（hook / plugin / middleware…）可直接用，別自造；以專案實際框架為準，非限定特定技術棧。
5. **已裝的依賴能解決嗎？** 掃專案既有依賴清單（`package.json` / `requirements.txt` / `go.mod` / `Cargo.toml`，依語言而定）——常常已經有能用的。
6. **真要新增，用現有依賴一行搞定嗎？** 傾向最小 diff。
7. **一行實在不行：寫最少能工作、不過度設計的代碼。** 見 `impl-author` 的「最小範圍」與 `design-patterns` / `refactoring` 的「對症才用、不為套而套」。

> **與 plan 的分工**：rung 3-6 是「**引不引**新依賴」的 author-time 自問；真要引一個全新依賴時，選型（列 ≥3 候選比較）走 `skills/plan/SKILL.md` §2 的「**引哪個**」。**若 plan 階段已為此任務做過套件評估，rung 3-6 不重開那個決策**，只處理 plan 未觸及的細節選擇。
