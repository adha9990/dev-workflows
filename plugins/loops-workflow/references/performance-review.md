# 效能審查（performance-review）

> performance-reviewer 用：審可預見規模下的退化。先過證據門檻避免臆測 finding，再看查詢 / 索引 / 分頁 / I/O 放大。

## 一、四件式證據門檻（先過這關）

報任何效能 finding 前，要能講齊四件，否則降 Non-blocking 或不報：

1. **觸發操作**：什麼使用者 / 系統操作會走到這條路。
2. **資料量假設**：在多大規模下會痛（明寫「在 N 筆 / M 並發下」）。
3. **路徑位置**：是熱路徑（列表 / 搜尋 / 每次互動）還是冷路徑（一次性 / 後台）。
4. **使用者後果**：會慢成什麼樣、卡住什麼操作。

只能說「可能變慢」、講不出上面四件的，不是可行動 finding。

## 二、查詢計畫退化訊號

對熱路徑的大集合查詢，看執行計畫（`EXPLAIN` / `EXPLAIN ANALYZE` 一類），找具體退化訊號：

- **全表掃描**（該走索引卻 scan 整張表）。
- **為排序 / 分組建臨時結構**（沒有可用的有序索引）。
- **join 內層退化成掃描**（巢狀迴圈每筆都掃）。
- 把抽象的「N+1 / 整表掃描」變成「真的去看計畫、認得退化訊號」。

## 三、索引設計

- **複合查詢要把 WHERE / ORDER BY / JOIN / LIMIT 一起設計索引**（含欄位順序），別只補單欄索引卻對不上實際多條件查詢。
- 考慮 **covering index**（查詢只讀索引就夠、不回表）。
- 加索引的寫入成本 / 空間成本是否值得。

## 四、分頁與大集合

- **穩定排序鍵**：有 LIMIT 但排序不穩定 → 翻頁會重複 / 漏項。
- 大集合仍用 offset 分頁且未評估成本 → 深頁昂貴；評估 keyset / cursor 分頁。

## 五、熱路徑 I/O 放大

- 熱路徑（列表 / 搜尋 / 匯入 / 每次 render）對**每一筆**做檔案系統探測 / 外部呼叫 / 大量小 I/O → 放大成 N 倍。
- 單一容器 / 目錄 / 集合**無界成長**，後續操作隨之線性 / 平方退化。
- 主要互動流上別放重活（同步阻塞、大量計算），該移到背景 / 延遲 / 分批。

## 六、Finding 寫法

雙視角：工程視角（哪條查詢 / 哪個迴圈、在多大規模退化、計畫證據），使用者視角（什麼操作在資料變多後會卡）。沒實測就標 `not measured`，不要寫沒跑出來的數字。
