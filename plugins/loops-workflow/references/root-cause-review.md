# 根治性審查（root-cause-review）

> root-cause-reviewer 用（條件式，bug fix 才派）：以 fresh context **預設這個修法只壓症狀，直到作者證明它修了病根**。這是獨立複查，補的正是「作者自己以為修了根因、其實沒有」的盲點。

## 一、症狀 vs 病根

判斷修法是治病根還是壓症狀：

- 在**症狀出現的下游**加 `try/catch` 吞掉 / 加 guard / `setTimeout` / retry / 重新整理來掩蓋，而沒有處理**上游真正製造壞狀態的地方** = 壓症狀。
- 病根修法要動到「第一個把合法狀態變成非法狀態」的那一層。

## 二、因果鏈最小事實框架

bug fix 應能交代（PR body 或改動本身要看得出）：

- **Reproduction**：怎麼重現。
- **Trigger**：什麼條件觸發。
- **Observed / Expected**：錯誤行為 vs 應有行為。
- **Causal chain**：哪一層、哪一步**第一次**把合法狀態變成非法 —— 這是病根定位。
- **Why this fix layer**：為什麼修在這一層（而不是更上游 / 更下游）。
- **Verification**：怎麼驗證修好了。

交代不出因果鏈、只看到「在出錯的地方擋一下」= 高度可疑為症狀 patch。

## 三、同類入口掃描

同一個病根往往有多個入口：

- 同一 service / 資料存取點的**其他 caller**會不會也踩到同一條因果鏈。
- 同一 parser / 正規化 / 轉換邏輯的**其他入口**是否一併修到。
- 只修了觸發 bug 的那一條路徑、漏了同類入口 → 換個操作就復發。

## 四、回歸測試合格標準

bug fix 的回歸測試要過五問（前四問對齊 `test-rubric.md` §7、第五問為本軸加強；缺 regression guard ≥P2，核心 / 高復發升 P1）：

- **能精確重現原 bug**（不是測個無關的近似情境）。
- 哪裡證明**已覆蓋**到那條路徑。
- **撤掉這次 fix，測試會失敗** —— 這是它真的守住病根的證明。
- 用「**接近但錯誤的實作**」會不會紅（防只測到表象、沒測到真行為）。
- **斷言正確行為**，而不只是「不 crash / 不報錯」。

## 五、分工與 Finding 寫法

不重述 `correctness-review` / `processing-reliability-reviewer` 的冪等 / 狀態流細節（必要時引用），本軸聚焦：症狀 vs 病根、因果鏈、同類入口、回歸測試撤 fix 必紅。雙視角：工程視角（病根在哪層、為何現修法只壓症狀），使用者視角（什麼操作下這個 bug 會以原樣或變形復發）。
