# 任務模板 + 該再拆四訊號

> plan 階段拆任務用。每個任務都要能**獨立 verify**，所以 Verification 欄必須是能實際跑的指令。

## 任務模板

```markdown
### 任務 <N>：<簡短標題，不含 "and">

- **Description**：<這個任務做什麼，一兩句>
- **Acceptance**：<做完的驗收條件，≤3 條；超過代表該拆>
  - [ ] <條件 1>
  - [ ] <條件 2>
- **Verification**：<能複製貼上去跑的具體指令 + 預期結果>
  - 例：`npm test -- src/foo.test.ts` → 全綠
  - 例：`curl -s localhost:3000/api/x | jq .total` → 回 42
- **Dependencies**：<依賴哪些前置任務 / 外部條件>
- **Files**：<會建 / 改哪些檔（精確路徑）>
- **Scope**：<明確不碰什麼，避免越界>
```

## 「該再拆」四訊號（命中任一就切小）

| 訊號 | 為什麼要拆 |
|------|------|
| 預估 **> 2 小時** | 太大，紅綠循環拉太長、難回頭 |
| **Acceptance > 3 條** | 一個任務扛太多責任，沒法乾淨驗收 |
| 跨 **2+ 子系統** | reviewer 沒法獨立接受 / 退回單一任務 |
| 標題裡有 **"and"** | 「做 A and B」就是兩個任務 |

## 尺寸與依賴

- 理想任務 = 一個紅綠循環能收掉的最小單位。
- 畫**依賴圖**：標出哪些任務可並行、哪些有先後。
- 每 **2–3 個任務**插一個 checkpoint（停下對齊，避免一路偏移）。
