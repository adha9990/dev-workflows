# 任務模板 + 該再拆四訊號

> plan 階段拆任務用。每個任務都要能**獨立 verify**，所以 Verification 欄必須是能實際跑的指令。

## 任務模板

```markdown
### 任務 <N>：<簡短標題，不含 "and">

- **Description**：<這個任務做什麼，一兩句>
- **Acceptance**：<做完的驗收條件，≤3 條；超過代表該拆>
  - [ ] <條件 1>
  - [ ] <條件 2>
  > Acceptance 條件**引用對應的 GWT 場景 ID**（`滿足 S1、S2`），讓 `場景 ID → task → 測試名（test_S1_…）→ verify 閘` 串成一條可追溯的線（SDD traceability）。無場景的內部任務照舊寫 pass/fail。
- **Verification**：<能複製貼上去跑的具體指令 + 預期結果>
  - 例：`npm test -- src/foo.test.ts` → 全綠
  - 例：`curl -s localhost:3000/api/x | jq .total` → 回 42
- **Dependencies**：<依賴哪些前置任務 / 外部條件>
- **Files**：<會建 / 改哪些檔（精確路徑）>
- **Scope**：<明確不碰什麼，避免越界>
```

## 「該再拆」四訊號（命中任一就切小）

| 訊號 | 為什麼要拆 |
|------|------|
| 預估 **> 2 小時** | 太大，紅綠循環拉太長、難回頭 |
| **Acceptance > 3 條** | 一個任務扛太多責任，沒法乾淨驗收 |
| 跨 **2+ 子系統** | reviewer 沒法獨立接受 / 退回單一任務 |
| 標題裡有 **"and"** | 「做 A and B」就是兩個任務 |

## 尺寸與依賴

- 理想任務 = 一個紅綠循環能收掉的最小單位。
- **垂直切片、不橫切**：別「全 DB → 全 API → 全 UI」橫著做；每個任務切成**端到端能跑的薄垂直切片**，做完系統還是 runnable —— 這樣每 2–3 任務的 checkpoint 才真的驗得到。
- **排序 risk-first**：相依允許下，**最高不確定性 / 最可能卡住的切片先做**，早點撞 blocker，別等沉沒成本堆高才發現（對齊 `AGENTS.md` 規則 10 fail-fast）。
- **尺寸階梯**：XS（1 檔 / 單一函式）· S（1–2 檔）· M（3–5 檔 / 單一子系統）· L（5–8 檔 / 多元件）· **XL（8+ 檔 / 跨子系統）= 一定要拆**。配上面四訊號一起用。
- 畫**依賴圖**：標出哪些任務可並行、哪些有先後。
- 每 **2–3 個任務**插一個 checkpoint（停下對齊，避免一路偏移）。
