# 交互閉環審查（ui-interaction-review）

> frontend-ui-reviewer 用：在元件結構 / 樣式之外，審「使用者操作後，資料是不是真的對」。核心不是好不好看，是**會不會假成功、會不會被舊回應蓋掉、相關視圖會不會沒同步**。

## 一、交互閉環

沿一條使用者操作追完整迴路，不要只看 UI 那一截：

```
使用者動作 → UI 即時表現 → 是否真的送出寫入請求 → 成功 / 失敗 / 重試 / 取消後的狀態 → 所有相關視圖是否回到同一真相
```

整條都成立才算閉環；斷在任一段都是缺口。

## 二、真實寫入 vs 假成功（最高風險）

- UI 顯示成功，但**寫入請求根本沒送出** = 假成功，按高風險（P1）看。
- **樂觀更新（optimistic）**：失敗時是否回滾到先前狀態、失敗是否對使用者可見、會不會殘留樂觀狀態假裝成功。只驗成功路徑、沒驗失敗回滾 = 信心不完整。
- 凡「看起來成功但沒有真實寫入證據」的互動，預設按高風險審。

## 三、衍生視圖 / 快取同步

- 一次寫入後，**所有衍生視圖 / 快取**是否同步？（不只當前清單，還有計數、標籤、群組、詳情、搜尋結果等任何反映同一資料的地方）
- 某個面板 / 區塊沒刷新 → 局部 stale，使用者看到自相矛盾的畫面。
- 快取失效 / 重新查詢的 key 是否涵蓋所有受影響的查詢。

## 四、並發 / 競態

- **重複點擊**：pending 中再次觸發，會不會重複提交 / 產生兩筆。
- **慢網**：載入中的中間態是否正確、會不會卡住或顯示錯誤的暫態。
- **亂序回應**：連續發出的請求晚回的覆蓋早回的，導致顯示舊值。
- **取消後殘留**：切走 / 取消後，殘留的 pending 回來是否還會寫進已不相關的畫面。

## 五、編輯 flush 與遺失

- debounce / 延遲保存的編輯：blur、切換目標、元件卸載時是否 **flush（保存）或明確丟棄**，而不是默默遺失。
- 切換編輯目標時，A 的未保存值會不會短暫顯示到 B。
- 保存失敗時是否保留使用者輸入，而非清空。

## 六、清單 / 選取 / 極端資料

- **虛擬化清單**：selection / focus / 右鍵目標應綁**資料 identity**，不綁 row index —— 滾動 DOM 復用會讓高亮 / 操作落到錯的項。
- 多選批量操作後，選取態是否一致。
- **極端資料下的交互正確性**（與效能不同軸）：超長文字、極多關聯、壞資源、極大資料量下，選取 / 開啟 / 預覽等操作是否仍可預測、不錯亂。
- **空 / 零資料態不得是 dead-end**：清單 / 管理 / 挑選頁在**完全沒有資料**時（新 workspace、全新帳號、空清單），主要的建立 / 新增入口（新增第一筆、primary create-affordance）**必須仍可達** —— 不能把空訊息（「尚無 X」）連同整個操作面板 / 建立按鈕一起藏掉，否則使用者根本無法建立第一筆（dead-end）。特別留意「空狀態 early-return 只渲染一句空訊息、跳過整個含建立入口的正常版面」這個常見反例。**空狀態的測試要斷言「建立入口仍在 / 仍可建第一筆」，不只「空訊息有出現」**。
- **使用者內容進 DOM 的 sink（安全交集）**：把使用者 / 外來內容塞進 `innerHTML` / `dangerouslySetInnerHTML` / `v-html`，或 `href` / `src` 帶 `javascript:` 之類 scheme —— **框架的自動 escape 只擋文字插值、不擋這些 sink**，要顯式 escape / sanitize。這同時是安全面（見 `security-checklist`）。

## 七、Finding 證據門檻

每筆 finding 要寫出 **什麼操作 → 什麼網路 / 並發條件 → 使用者實際看到什麼錯**。純「可能有競態」講不出觸發情境的，降 Non-blocking note。
